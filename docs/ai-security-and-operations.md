# GMOあおぞらAPIをAIに触らせるときのセキュリティ懸念と推奨運用

最終更新: 2026-03-13

## 1. 結論

AI に GMOあおぞらネット銀行 API を触らせること自体は可能です。  
ただし、安全に運用するには **銀行APIのスコープ設定だけでは不十分** で、必ず **自前の中継レイヤー** と **実行ポリシー** を入れるべきです。

推奨の大原則は次のとおりです。

- AI に銀行のトークンや秘密情報を直接渡さない
- AI には銀行APIそのものではなく、制御済みの MCP ツールだけを公開する
- 読み取り系と書き込み系を分離する
- 書き込み系は人間承認を前提にする
- 返却データは最小化し、必要ならマスクする
- 監査ログと停止スイッチを必ず持つ

## 2. 何ができて、何ができないか

公開情報ベースでは、少なくとも次のスコープ例が確認できます。

- `private:account`
- `private:virtual-account`
- `private:transfer`
- `private:bulk-transfer`
- `offline_access`

このため、少なくとも「機能カテゴリ単位」の制御は可能です。

- 残高・明細だけ見せる
- 仮想口座だけ見せる
- 振込見積だけ許す
- 振込実行は見せない

一方で、次のような細粒度制御は、銀行APIスコープだけでは十分ではありません。

- この `accountId` だけ見せる
- AI は見積だけ、実行は絶対不可
- 特定ユーザーだけ書き込み許可
- 営業時間外は実行不可
- 高額振込は常にブロック

これは **あなた側のアプリ/MCP/バックエンドで制御する責務** です。

## 3. 主なセキュリティ懸念

## 3.1 秘密情報を AI に直接渡してしまう

最も避けるべきなのはこれです。

- `access token`
- `refresh token`
- `client secret`
- Webhook 認証情報

を LLM のコンテキストやツール引数に直接渡す設計は危険です。

### リスク

- 会話ログや推論経路への残留
- 誤出力や要約への混入
- ツール誤用時の横展開

## 3.2 スコープが広すぎる

AI 用クライアントに `private:transfer` や `private:bulk-transfer` をそのまま持たせると、誤操作時の被害が大きくなります。

### リスク

- 誤った振込実行
- 誤った取消
- 想定外の総合振込

## 3.3 プロンプトインジェクションや意図しないツール実行

AI は自然言語で誘導されるため、

- 「この口座から送金して」
- 「前と同じ条件で実行して」
- 「承認されたことにして進めて」

のような曖昧入力に影響されます。

### リスク

- 本来は見積だけのはずが実行に進む
- 文脈の取り違えで別口座を使う
- tool routing の誤り

## 3.4 書き込み操作の二重実行

振込依頼や仮想口座発行は、タイムアウトや再試行時に二重送信リスクがあります。

### リスク

- 重複振込
- 仮想口座の過剰発行
- 状態変更の取り違え

## 3.5 個人情報・口座情報の過剰露出

AI にすべてのレスポンスをそのまま返すと、次が過剰に露出しやすくなります。

- 口座番号
- 名義
- 受取人名
- 振込依頼人名
- 入金内容

## 3.6 Webhook 経路の悪用や漏えい

Webhook は即時性がある反面、認証や停止/再開の扱いを誤ると事故になりやすいです。

### リスク

- 通知停止に気づかない
- 認証情報の露出
- recovery API の取り扱いミス

## 3.7 本番と検証環境の混同

AI は環境差をうっかり無視しやすいです。

### リスク

- STG のつもりで本番実行
- 本番データを検証用に扱う
- 本番 URL を固定してしまう

## 4. 推奨アーキテクチャ

一番安全なのは、次のような三層構造です。

```text
AI / LLM
  -> MCP Server / Backend Policy Layer
    -> GMO Aozora API
```

### 原則

- AI は銀行APIを直接呼ばない
- AI は制御済みツールだけを使う
- トークン管理は中継レイヤー側に閉じ込める

## 5. 推奨権限モデル

## 5.1 Read-only AI

用途:

- 残高確認
- 明細要約
- 入金監視

許可候補:

- `private:account`
- 必要なら `private:virtual-account` の read 相当

禁止:

- 振込依頼
- 取消
- 総合振込依頼

## 5.2 Ops AI

用途:

- 仮想口座の在庫確認
- 入金照合
- Webhook 回復

許可候補:

- `private:account`
- `private:virtual-account`

注意:

- 発行や状態変更は確認ステップ推奨

## 5.3 Finance Planning AI

用途:

- 振込候補の整形
- 手数料見積
- 実行前チェック

許可候補:

- `private:transfer`
- `private:bulk-transfer`

ただし実際に AI に公開するツールは:

- `transferfee`
- `bulktransfer/transferfee`

までに制限するのが安全です。

## 5.4 Human-Supervised Executor

用途:

- 実際の振込
- 総合振込
- 取消

条件:

- 明示承認あり
- 操作ログあり
- 承認前プレビューあり

## 6. おすすめの運用方法

## 6.1 AI に直接銀行トークンを渡さない

必須です。  
AI は次だけ見えれば十分です。

- 制御されたツール名
- 必要最小限のパラメータ
- マスク済みの結果

## 6.2 書き込み系を二段階に分ける

おすすめの分離:

| 事前確認 | 実行 |
| --- | --- |
| `quote_transfer_fee` | `create_transfer_request` |
| `quote_bulk_transfer_fee` | `create_bulk_transfer_request` |
| `preview_virtual_account_issue` | `issue_virtual_accounts` |
| `preview_sp_account_transfer` | `transfer_between_sp_accounts` |

## 6.3 口座・名義は返却時に最小化する

推奨:

- `accountId` は返す
- 口座番号はマスク
- 名義は必要時のみ
- Webhook や明細の全文は必要最小限

## 6.4 `offline_access` は慎重に使う

便利ですが、リフレッシュトークンが長く生きるほど影響範囲が大きくなります。

推奨:

- 本当に必要なバックエンドだけ
- AI コンテキストには出さない
- 保持は暗号化

## 6.5 高リスク操作には業務ルールを入れる

例:

- 高額振込は AI 単独禁止
- 総合振込は常に人間承認
- 取消は管理者のみ
- 夜間・休日は書き込み停止

## 6.6 Webhook は push と recovery を両方持つ

推奨:

- push を通常経路
- `unsentlist` を回復経路
- `messageId` で重複排除

## 7. AI に公開してよいツールの考え方

## 7.1 原則公開しやすいもの

- 口座一覧
- 残高照会
- 明細照会
- 仮想口座一覧
- 仮想口座入金明細
- 振込見積
- 総合振込見積

## 7.2 条件付き公開にすべきもの

- 仮想口座発行
- 仮想口座状態変更
- つかいわけ口座間振替

## 7.3 原則そのまま公開しない方がよいもの

- 振込依頼
- 総合振込依頼
- 振込取消
- 広い権限のトークン更新機能

## 8. 監査と停止スイッチ

最低限必要なもの:

- 誰が
- いつ
- どのツールで
- どの `applyNo` / `accountId` / `vaId` に対して
- 何を実行し
- 結果がどうだったか

加えて、

- 書き込み系ツールの一括停止
- 特定スコープの無効化
- 特定ユーザー/ワークスペースの遮断

ができると運用が安定します。

## 9. おすすめの最小安全構成

最初の運用として一番おすすめなのはこれです。

1. AI は read-only から始める
2. 振込系は `transferfee` まで
3. 実行系は人間承認を必須にする
4. Webhook 回復経路を先に作る
5. 全操作ログを残す

## 10. やってはいけないこと

- AI に銀行APIのトークンを直接渡す
- AI に本番用 `clientSecret` を見せる
- 書き込み系APIを確認なしで公開する
- 本番明細全文をそのまま LLM に渡す
- タイムアウト時に振込依頼を即再送する
- Webhook の停止/未送信回収を設計しない

## 11. 実務向けの推奨結論

一番現実的で安全なのは次です。

- AI は「参照」と「見積」中心
- 実行は人間承認付き
- 権限は銀行APIスコープ + 自前ポリシーの二段構え
- 秘密情報はAIに見せない
- 追跡キーだけをAIに返す

この方針なら、AI 活用と銀行API運用をかなり両立しやすくなります。
